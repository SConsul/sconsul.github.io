Multi Client Server

Using socket programming, I made a server capable of handling multiple active clients. When realising such servers, it is important to be able to be able to communicate with all the clients and process their instructions that do not have a fixed pattern. Furthermore, the accept and read system calls are 'blocking' (i.e. server comes to a standstill until it receives data). This makes handling of concurrent clients impossible. A naive way of handling this is to assign a new thread to every client. This, however, is not scalable and is prone to issues such as deadlock. A nifty work around is using non-blocking sockets and polling or event driven I/O system calls (epoll in Linux).
I have decided to use the event-driven I/O. What happens is that the server continuously montitors multiple file secriptors, waiting for one of them to become active. The active client is then binded just long enough to a child socket to complete the request. Meanwhile, the server is still able to monitor the other clients and schedule any incoming requests.
